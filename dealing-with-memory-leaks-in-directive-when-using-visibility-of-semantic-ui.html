Dealing with memory leaks in directive when using visibility of Semantic UI

I currently modified some code to enhance search experience at work for our site. Infinite scrolling can load more search results unknowingly during a search session, we adopted this way to get better search experience. However, unexpected memory leaks happened as developing.

Our frontend is made by AngularJS and Semantic UI. Simply describe how I did to implement infinite scrolling, just call visibility behavior of Semantic UI. I was following the example of visibility from official site, adding onBottomVisible callback into a Angular directive with restriction 'C' to trigger loading funtion for more search results. The code looks like this:

$element.visibility({
    once: false,

    // Update size when new content loads
    observeChanges: true,

    // Content on bottom edge visible
    onBottomVisible: function() {
        // loads more search results
    }
});

There are two Angular routes, main page and user page, both be able to call this directive by matching the class name. Main page is to search and display results from the whole data of site, user page do the same things from a user's personal data. The memory leaks happen when a user switches one page to anthor one, cause search results will be loaded twice or more with previous page data. But why? How to fix it?

Trying to figure out what happened there, I read the source code of the visibility from Semantic UI on GitHub. The version we were using is 2.1.8, I was trying to glimpse about observeChanges context in visibility source code. It seems that the destroy would not be called from it. Going forward to check the latest version 2.2.2. The function destroy could be run from observeChanges indirectly by event contextChanged when condition is satisfied. So I updated the package to 2.2.2, and then everything worked fine to me. The memory leaks were fixed.
